
# 프로그램의 목적
이 프로그램은 터미널에서 동작하는 Chef, Ansible같은 Configuration Management Tool이야.
간략한 python 파일 하나에 동작할 명세와 동작 스크립트가 통합된 구성으로 동작해.

# 지침 및 개요
- 코드 작성: 읽기 쉽고 유지보수하기 쉽게 간결하게 작성. 로직 코드는 최대한 유닛 테스트를 만들 수 있게 작성
- 코드 수정: 꼭 필요한 부분만 수정한다! 기존 프로젝트 컨벤션과 스타일을 엄격히 준수
- 함수화 및 재활용: 코드가 함수화에 적합하면 함수로 감싸고, 중복된 코드가 없게 기존 함수를 최대한 재활용
- 커밋: 절대 자동으로 커밋하지 말고!, 요청 시에만 이전 작업 내역 바탕으로 커밋 메시지 작성 후 수행
- 문서 수정: 문서도 꼭 필요한 부분만 최소한으로 수정하고, 수정후 수정된 내역을 보여줘.

## 정의 문서 목록
- guide.md: this document
- spec.md: 동작 구조와 스펙이 상세히 나열
- test.md: 유닛테스트와 docker를 이용한 실제 구동 테스트 목록

- manual.md: gat용 설정 파일 작성 매뉴얼 

### Spec문서
동작 구조와 스펙을 상세히 나열


### Test문서
모든 Spec문서는 각각 Test문서를 가지고 있는다. 이 파일에는 스펙의 모든 기능 및 추가적으로 작성 가능한 테스트 목록이 명세되어있다.
테스트 항목은 제일 앞에 체크박스로 작성 여부, 테스트 방법 정의 ( 파일위치 - 테스트 항목명) 으로 구성된다.
예시) - [x] Updated 컬럼: 오늘 수정한 이슈는 HH:mm 형식으로 표시되야 한다 (app/test/issue_list_date_test.dart - Format date for today)
테스트가 구현되면 체크박스를 체크해줘 [x]

- test.md: 유닛테스트와 docker를 이용한 실제 구동 테스트 목록


## 작업절차(꼭 절차에 따를것)
- 수정하기전에 먼저 변경된 명세를 스펙 문서(ai/*-spec.md)에 먼저 정리 한다.
  - 버그일 경우에는 스펙문서에는 기재하지 않아도 아래 테스트 문서(*-test.md)에 기재한다
- 이후에 꼭! 정의된 스펙문서를(*-spec.md) 기준으로 테스트 문서(*-test.md)에 테스트 항목을 추가하고 유닛 테스트 작성 수행
  - 테스트 함수 위치와 어떤식으로 테스트를 구현했는지 명세
  - 만약! 작성할수 없을때는 작성 불가 사유를 ai/*.md문서에 같이 기재한다. 

## 유닛 테스트
- 유닛 테스트 작성시 Application에서 서버 호출이 필요한 경우에는 호출함수에 가상 호출로 서버 호출을 가상으로 처리한다.

## 커밋
- 커밋 메시지 작성시 타이틀은 "<관련된대분류>: <관련된기능 혹은 페이지이름 - <수정내역>"으로 작성
  - 관련된 대분류는 app, ser, sbedit, sbdoc
  - 커밋 메시지 예시> app: mainWindow - display summary news
